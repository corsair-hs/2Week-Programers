메타인지
1. 뭘 모르는지, 아는지
학습은 어떻게보면 뭘모르는지 알아가는 과정
- 학사 : 이제야 좀 아는 게 생겼다고 자부함
- 석사 : 공부를 조금 더 해보니 모르는 게 있음
- 박사 : 이제 아는 것보다 모르는 게 훨씬 더 많음
- 교수 : 나도 내가 무슨 말을 하는지 모르는 상태
-> 모르는 게 더 많게 느낄수록 내가 성장하는 과정이니, 안심해라.
2. 뭘 알고 있는지, 모르는지 인지하는 게 중요
TIL = 오늘 알게된 것
WIL = 이번주에 알게된 것
배운거 까먹지 않게 일단 냅다 키워드라도 남기면서 쓰기
요구사항/문제사항 나눠서 쓰기
내가 아는 것을 진짜 아는 것으로 유지하기 위해, TIL/WIL 정리하기



2. 개발공부는 왜 이렇게 힘든지?
(듣도보도 못한 단어들 천지, 약어는 또 왜이렇게 많은지, 표현도 추상적, 익숙해질 때 쯤 새로운 기술/버전도 나오고. 심지어 분야도 너무 많음)

내가 하는 일이 힘들수록 아무나 할 수 없는 더욱 가치있는 일 이라는 증거입니다.

즉, 개발공부는 힘들기 때문에 더욱 가치 있는 것이다.



3. 열심히 하는 사람, 잘하는 사람보다 더 잘 학습하는 법
모든 것들을 계획하고 실행한 문동은 -> 어떻게 가능할까? 더 똑똑했기 때문일까?
-> 문동은의 눈빛 : 어떤 감정? 악에 받친 사람은 두려움, 막막함, 피곤, 귀찮음 느끼지 못함 -> 오롯이 해내고 말겠다는 마음 뿐.

문득 포기하고 싶을 때,
이를 악물고 문동은의 악에 받친 감정과 눈빛을 기억하라. 무엇이든 해내고 말거다.


4. 번아웃은 언제 올까?
해야할일이 너무 많을 때? 공부해야 할게 너무 많을 때?
번아웃은 오늘 할일을 감당할 수 없을 때, 오늘공부해야할 내용을 가늠할 수 없을 때 옵니다.
내가 지금 당장, 해야할 일을 봤을 때 뭐부터 해야할지 모르는 상황이 오면 그 때 번아웃이 온다.
오늘 공부할 내용은 오늘 끝내야 한다. 그래야 번아웃이 오지 않음

번아웃 오지 않는 방법
오늘 무리하지 않는 것이 아니라
내일 무리하지 않도록 오늘 해냄으로서 
내일의 번아웃을 방지하는 것이다.




둘쨰. 어떻게 적응할까?
01. 삼성SDS 현직 개발자가 알려주는 신입이 적응하는 법
- 질문의 적정성을 잘 찾아서 적극적으로 질문하며 업무에 임하는 자세가 중요.
  즉, 이렇게 되기 위해선 내가 뭘알고 뭘 모르는지 알아야함

02. 네이버 현직 개발자가 알려주는 신입이 적응하는 법
- 적극적으로 물어볼 것
- 겸손해야 할 것
- 실수를 무서워 하지 않는 과감함이 있으면 좋겠음
- 똑같은 거 여러번 묻지 않도록 기록하는 습관이 있어야 함
  (TIL / WIL 작성해본 경험이 도움될 것이다.)

03. 테슬라 CEO 일론 머스크가 알려주는 신입을 뽑는 기준
- 살아온 과정을 얘기해달라고 함
  - 그 과정에서 내렸던 결정들과 그 이유
  - 직접 겪었던 문제가 무엇
  - 어떻게 해결했는지
  - 문제를 진정으로 해결해봤던 사람은 어떻게 해결했는지, 디테일까지 다 알고있음
  - 문제를 해결한 척 하는 사람은 어느순간 막힘
- 문제를 진정으로 해결하고 어떤 문제였는지 어떻게 그것을 해결했는지 정확히, 세부사항까지 말할 수 있어야 함 -> 더 큰 문제도 해결할 수 있음

이 번 항해캠프에서 이러한 문제들을 정말 많이 만들어보고, 이 것에 대해서 해결을 직접 해보아야 함.



개인의 메타인지를 높이는 방법
- 내 업무의 핵심질문을 정의
- 내가 맡은 업무의 배경을 조사하고,
그 배경을 만들어낸 원인이 무엇인지 파악 (좀 더 큰 그림을 그려라)
- 업무의 주체에 대해서 트리형태로 분류하는 연습

조직의 메타인지력을 높이는 방법
- 조직의 메타인지력이 낮으면 이직률이 높음
- 내가 CEO라면 이 문제에 대해서 어떤 결정을 내릴까?
- 내가 동료들 또는 조직의 목표를 알고있고 함꼐하고 있는가?
- 문제를 바라보는 관점이 팀원마다 어떻게 다른지 정리



셋째. 어떻게 일해야 할까?
01. 직장을 다니는 개발자가 아닌, 직업을 가지는 개발자가 되자.
- 직장: 회사원으로써 내가 주어진 일을 해내는 곳
- 직업: 회사에서 내가 업무의 전문성을 키우는 과정

02. 진심으로 개발하는 개발자가 되자.
- 진심으로 개발하는 것이란
기능이 아닌 서비스를 개발하는 사람
그냥 개발자 : 맡은 기능을 잘하는 개발
유능한 개발자: 맡은 서비스를 잘 개발하는 개발자
기능 : 하나의 나무들과 같음 -> 이것을 잘하면 연봉을 받고
서비스 : 하나의 산/숲 -> 이것을 잘하면 보너스를 받음
